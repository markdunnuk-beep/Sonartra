# Admin Users Completed-Result Fixture

## Fixture Added

The repo now includes a deterministic local QA fixture seeded by:

- `npm run seed:admin-users-fixture`

It creates one stable internal user:

- `qa-completed-user@sonartra.local`
- `QA Completed User`

## Records Created

The fixture creates and maintains one canonical completed chain for `wplp80`:

- one `users` row
- one `user_assessment_assignments` row with `status = completed` and `order_index = 0`
- one linked `attempts` row with `lifecycle_status = RESULT_READY`
- one linked `results` row with a canonical persisted payload generated by the real engine
- one full set of persisted `responses` rows for the seeded attempt

Result linkage remains:

- `assignment -> attempt -> result`

## Completed-Result Coverage

This fixture keeps the Admin Users completed branch inspectable through normal persisted reads:

- `/admin/users` shows the fixture user as `Completed`
- `/admin/users/[userId]` shows a historical completed row
- the detail timeline exposes the canonical result link
- the completed row remains history-locked under the existing assignment mutation rules

## Local QA Use

Recommended local sequence:

1. `npm run db:migrate`
2. `npm run seed:wplp80`
3. `npm run seed:admin-users-fixture`

The fixture seed validates the persisted list/detail read models before it commits and prints the admin detail path plus result path for local inspection.

## Scope

This fixture is intended for local and non-production QA only. The seed script refuses to run when `NODE_ENV=production`.

## Maintenance Caution

- Keep the fixture on the existing canonical schema only.
- Do not add fixture-only result shortcuts or UI-side logic.
- If the `wplp80` published version changes materially, reseed the fixture so its persisted responses and canonical payload stay aligned.
