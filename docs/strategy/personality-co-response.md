# Sonartra Response to Personality.co

Date: 2026-04-18  
Source reference: [docs/competitor-analysis/personality-co-deep-dive.md](/C:/Projects/sonartra-build/Sonartra/docs/competitor-analysis/personality-co-deep-dive.md)

## Purpose

This document converts the Personality.co competitor teardown into explicit Sonartra product, UX, pricing, and trust decisions.

Sonartra is not a generic personality-test subscription product.  
Sonartra is a premium behavioural intelligence and business performance platform.

The response strategy is therefore:

- adopt only what improves clarity and completion
- adapt what can be upgraded into premium, business-grade UX
- reject anything that weakens trust, rigour, or enterprise positioning

## Competitor Pattern Decisions

| Competitor pattern | Decision | Sonartra response |

| Simple, clear homepage CTA into assessment | `Adopt` | Keep a direct, visible primary CTA into the flagship assessment flow |

| Consumer self-discovery headline | `Reject` | Sonartra must lead with business performance, behavioural intelligence, and decision quality |

| Five-point Likert question interaction | `Adopt` | Use a clear, consistent answer mechanic with minimal interaction ambiguity |
| Multi-step progress with percent complete | `Adopt` | Show percent complete, section progress, and completion confidence throughout |
| No account wall before answering questions | `Adapt` | Allow low-friction start, but capture identity only when it supports continuity and trust |
| Hard answer requirement before continue | `Adopt` | Require complete response sets for a valid result, consistent with the engine contract |
| End-of-test completion confirmation | `Adopt` | Keep a completion handoff state before results |
| Reveal a simplified type before payment | `Reject` | Sonartra must not cheapen the output into a teaser-type reveal |
| Gate the real insight behind a post-test paywall | `Reject` | Sonartra should not create sunk-cost resentment or trial-funnel optics |
| Low-cost trial that auto-renews | `Reject` | This conflicts with premium trust and enterprise credibility |
| Repeated sales blocks and conversion pressure | `Reject` | Sonartra should use restrained persuasion and strong product substance instead |

## Sonartra Equivalents

## Homepage Positioning

### Sonartra decision

The homepage must position Sonartra as a serious behavioural intelligence platform for workplace performance, not as a self-discovery quiz.

### Required homepage message structure

1. Hero
   - Primary message: Sonartra helps leaders and teams understand the behavioural patterns shaping performance.
   - Primary CTA: `Start Sonartra Signals`
   - Secondary CTA: `View Platform`

2. Immediate proof layer
   - What Sonartra measures
   - Who it is for
   - Why the output is useful in work settings

3. Model clarity layer
   - Explain signals, domains, and deterministic outputs
   - State that outputs are structured, rule-based, and built for action

4. Use-case layer
   - leadership self-awareness
   - collaboration and communication
   - team effectiveness
   - development focus

5. Report/output layer
   - show overview, strengths, watchouts, development focus, and manager/team application

6. Trust layer
   - methodology summary
   - implementation standards
   - pilot outcomes or case evidence

### Homepage copy direction

Use:

- behavioural intelligence
- signal clarity
- decision patterns
- leadership insight
- team effectiveness
- business performance

Avoid:

- find out who you truly are
- exact personality type
- life-changing insights
- soft self-help language
- catalogue bundle language

## Test Onboarding

### Sonartra decision

Sonartra should keep onboarding short, but it must feel more intentional and business-relevant than Personality.co.

### Sonartra onboarding screen should include

- assessment name
- expected duration
- what the assessment measures
- how to answer well
- completion expectation
- privacy/trust note

### Recommended onboarding structure

- Heading: `Prepare for Sonartra Signals`
- Subtext: explain that the assessment identifies behavioural operating patterns relevant to leadership, collaboration, and performance
- Guidance bullets:
  - answer based on typical behaviour, not ideal behaviour
  - complete the full assessment in one sitting where possible
  - there are no right answers
  - every answer contributes to the final result
- CTA: `Begin assessment`

### What Sonartra should not do

- no hype countdowns
- no pseudo-analysis animation before the user has earned trust
- no language implying magic accuracy or hidden processing

## Question UX

### Sonartra decision

The runner should stay minimal, stable, and focused. The question experience should feel like a premium work tool, not a consumer quiz.

### Recommended question UX

- one clear question block per screen state
- stable answer scale across all questions
- keyboard and pointer friendly
- no decorative clutter
- explicit selected state
- section label visible
- optional review access before final submission

### Answer mechanic

Adopt the clarity of a five-point scale, but tune the labels for Sonartra’s tone. Example:

- Strongly unlike me
- Somewhat unlike me
- Mixed
- Somewhat like me
- Strongly like me

This keeps the interaction simple while avoiding overly generic survey language.

### UX design notes

- questions should read behaviourally, not typologically
- the interface must not imply scoring logic in the UI
- the runner must remain a delivery layer only

## Progress System

### Sonartra decision

Use explicit progress, but make it more useful than Personality.co's flat percentage system.

### Sonartra progress model

Display:

- overall percentage
- current section
- question count
- completion state

Recommended example:

- `42% complete`
- `Section 4 of 8`
- `Question 34 of 80`

### Progress principles

- progress should reduce uncertainty
- progress should never pressure or manipulate
- users should know whether they can review prior answers
- incomplete state should be handled clearly and calmly

## Completion Experience

### Sonartra decision

Completion should feel like a professional handoff, not a sales hinge.

### Sonartra completion state should do

- confirm completion
- explain that results are being finalized
- reinforce what the user will receive
- offer a controlled edit/review path if appropriate

### Recommended completion structure

- Heading: `Assessment complete`
- Subtext: `Your responses have been submitted. Sonartra is finalising your behavioural intelligence report.`
- Secondary information:
  - result sections included
  - expected availability state
  - where the result will appear
- Actions:
  - `View results` when ready
  - `Review responses` only if reopening is allowed by product rules

### What Sonartra should avoid

- no bait-style teaser copy
- no personality-type preview
- no commercial interruption between completion and core result access if the user is already entitled to the report

## Results Reveal

### Sonartra decision

Results should open with signal clarity and business application, not a single dramatic label.

### Sonartra results reveal should show first

1. behavioural overview
2. top signal or dominant pattern cluster
3. ranked signals
4. domain summary
5. strengths
6. watchouts
7. development focus
8. manager/team application

### Recommended first-screen result composition

- result status: `Ready`
- report title: `Your Sonartra Signals Report`
- overview summary
- top behavioural patterns
- confidence/coverage note if needed
- CTA into deeper sections

### Results framing rule

Sonartra should explain the user, not label the user.  
The report must feel actionable, structured, and work-relevant from the first screen.

## Pricing Model

### Sonartra decision

Sonartra should use a transparent premium pricing model with no low-trust trial mechanics.

### Pricing requirements

- pricing must be understandable before commitment
- no hidden renewal surprises
- no tripwire-to-subscription funnel
- no post-test emotional leverage pricing
- commercial model must reinforce premium trust

### Recommended pricing approach

For MVP, Sonartra should use one of these two models:

1. Premium assessment purchase
   - a clear one-off price for the individual report
   - includes full report access and any promised follow-up assets

2. Team or organisational package
   - clearly scoped pricing via contact/demo
   - positioned around deployment, reporting, and business use

### Recommended rule

If an individual user can start an assessment without paying first, Sonartra must still make result entitlement and pricing logic clear before completion.  
Do not let users finish a serious business assessment only to discover a surprise paywall.

### Explicitly rejected pricing patterns

- `GBP 1.95` entry trials
- auto-renew after a short access window
- recurring billing hidden in fine print
- dashboard cancellation as the main trust repair mechanism

## Trust Positioning

### Sonartra decision

Trust must be earned through precision, consistency, and restraint.

### Sonartra trust stack

- clear model explanation
- precise claims
- deterministic output language
- visible business use cases
- transparent pricing
- credible company and support details
- verifiable proof over theatrical proof

### Sonartra should say

- what the assessment measures
- how outputs are generated at a high level
- how results should be used
- what the assessment is not intended to do

### Sonartra should not say

- exact personality type
- scientifically proven if that claim cannot be defended precisely
- transformative promises without evidence
- expert-created as a substitute for actual model transparency

## Five UX Principles Sonartra Must Not Violate

1. `No hidden commercial switch`
   The user must not discover core pricing logic only after finishing a long assessment.

2. `No contradiction between promise and proof`
   Sonartra must never use strong validation language in one place and soft disclaimers that undermine it elsewhere.

3. `No generic personality theatre`
   The UX must stay anchored in behavioural intelligence and business application, not consumer typing tropes.

4. `No feature sprawl in the core journey`
   The flagship assessment and result experience must not be diluted by unrelated libraries, courses, badges, or filler add-ons.

5. `No UI-side meaning-making`
   The interface may explain results, but it must not invent or compute them. The engine remains the single source of truth.

## Final Sonartra Position

Personality.co shows that simple assessment UX and disciplined progress design can increase completion. Sonartra should adopt that clarity.

Personality.co also shows the downside of consumer-funnel logic: vague scientific framing, bundle sprawl, aggressive conversion mechanics, and subscription-trial optics. Sonartra should reject that model completely.

The correct Sonartra response is:

- simple but premium
- clear but rigorous
- commercially transparent
- business-relevant from first impression to final report
- fully aligned with one engine, one scoring path, and one canonical result experience
